# Слоты выгрузки и жизненный цикл конфигураций

Каждая конфигурация 1С в conf-doc имеет **постоянный слот выгрузки** на диске сервера:

```
output/exports/{ИмяКонфигурации}/
```

Имя берётся из `Configuration.xml` → `Properties/Name` (например `БухгалтерияПредприятия`).

Параллельно в SQLite и на диске хранятся производные данные:

| Путь | Содержимое |
|------|------------|
| `output/exports/{Имя}/` | XML-выгрузка (слот) |
| `output/docs/{Имя}/` | Markdown по объектам |
| `output/vectors/{Имя}/` | FAISS-индекс эмбеддингов |
| `output/db/metadata.db` | Метаданные, чанки, кэш эмбеддингов (все конфигурации) |

**Staging** (`data/export/` в Docker → `/data/export` в контейнере) — опциональная read-only папка с выгрузкой. Её не обязательно копировать в слот: по умолчанию слот **привязывается** к пути через файл `.import_source`.

---

## Веб-UI: вкладка «Конфигурации»

Откройте `http://localhost:8050/` (Docker) или `http://localhost:8000/` (локальный `conf-doc serve`).

### Колонки таблицы

| Колонка | Значение |
|---------|----------|
| **Выгрузка** | `staging` — слот привязан к пути без копирования; `в слоте` — файлы лежат в `exports/{Имя}/`; `битая выгрузка` / `нет выгрузки` — нужен импорт |
| **Индексация** | Дата последнего прохода парсера (объекты и чанки в SQLite) |
| **Эмбеддинги** | Модель FAISS или `нет индекса` — нужна переиндексация |

### Кнопки в строке конфигурации

| Кнопка | Действие | API |
|--------|----------|-----|
| **Обновить** | Парсинг XML из слота → markdown и чанки в SQLite. **Без** вызова API эмбеддингов | `POST /configurations/{name}/index` с `skip_embeddings: true` |
| **Переиндексировать** | Только эмбеддинги и FAISS по чанкам, уже лежащим в базе. **Без** повторного обхода XML | `POST /configurations/{name}/embed` |
| **Удалить** | Удаление из БД и с диска (markdown, FAISS, папка слота) | `DELETE /configurations/{name}` |

Кнопки блокируются, пока по этой конфигурации идёт фоновая задача. Над таблицей отображается строка прогресса; подробные логи — на вкладке **«Логи»**.

**Обновить** активна, если в слоте есть выгрузка (`has_export`).  
**Переиндексировать** активна, если конфигурация уже в базе **и** есть выгрузка в слоте.

Если конфигурация есть в базе, но **нет выгрузки в слоте** (колонка «Выгрузка» — `нет в слоте`), сначала импортируйте выгрузку через мастер **«+ Новая»**.

### Мастер «+ Новая»

Используйте для **первичного** добавления конфигурации или полного цикла (импорт → проверка → эмбеддинги → индексация).

1. **Имя** — подставляется из `Configuration.xml` при указании пути или ZIP.
2. **Выгрузка в слот** — ZIP или путь (staging).
3. **Проверка** — `POST .../detect`, сверка имени с XML.
4. **Эмбеддинги** — провайдер и модель (сохраняются в `config.yaml` для этой конфигурации).
5. **Запустить индексацию** — полный цикл с эмбеддингами (аналог «Обновить» + «Переиндексировать» за один раз).

## Добавление конфигурации в слот

### Вариант A: путь на сервере (Docker staging)

1. Положите XML-выгрузку на хост, например в `data/export/` (в контейнере это `/data/export`).
2. Убедитесь, что путь разрешён в `config.yaml`:

   ```yaml
   import_roots:
     - /data/export
   ```

3. **«+ Новая»** → вкладка **«Путь (staging)»** → укажите `/data/export`.
4. **«Привязать путь к слоту»** — создаётся `output/exports/{Имя}/.import_source` с путём к выгрузке (быстро, без копирования гигабайтных каталогов).
5. **«Проверить слот»** — должно быть зелёное OK.
6. **«Обновить»** (метаданные) → **«Переиндексировать»** (эмбеддинги).

Чекбокс **«Скопировать файлы в слот»** (`mirror: true` в API) — физическое копирование в `output/exports/{Имя}/`. Нужен редко (перенос с staging, отвязка от исходного пути). На Docker + Windows томе большие выгрузки копируются медленно и ненадёжно — предпочитайте привязку.

### Вариант B: ZIP-архив

1. **«+ Новая»** → вкладка **«ZIP-архив»** → выберите `.zip` с выгрузкой.
2. **«Импортировать ZIP в слот»** — распаковка в `output/exports/{Имя}/`.
3. Далее — проверка и индексация, как выше.

### Вариант C: API / CLI

```bash
# Зарегистрировать слот (создать папку)
curl -X POST http://localhost:8050/configurations \
  -H "Content-Type: application/json" \
  -d '{"name":"БухгалтерияПредприятия"}'

# Привязать путь (по умолчанию без копирования)
curl -X POST http://localhost:8050/configurations/БухгалтерияПредприятия/import-path \
  -H "Content-Type: application/json" \
  -d '{"source":"/data/export","mirror":false}'

# Проверка
curl -X POST http://localhost:8050/configurations/БухгалтерияПредприятия/detect

# Только метаданные и чанки
curl -X POST http://localhost:8050/configurations/БухгалтерияПредприятия/index \
  -H "Content-Type: application/json" \
  -d '{"skip_embeddings":true}'

# Только эмбеддинги
curl -X POST http://localhost:8050/configurations/БухгалтерияПредприятия/embed \
  -H "Content-Type: application/json" \
  -d '{"force":false}'
```

CLI:

```bash
conf-doc index --configuration БухгалтерияПредприятия --skip-embeddings
conf-doc embed --configuration БухгалтерияПредприятия
```

## Несколько конфигураций

У каждой конфигурации — **свой слот** и (рекомендуется) **свой staging-путь**:

```
/data/export-bp   →  БухгалтерияПредприятия
/data/export-zup  →  ЗарплатаИУправлениеПерсоналомКОРП
```

`docker-compose.yml` (фрагмент):

```yaml
services:
  conf-doc:
    volumes:
      - ./data/export-bp:/data/export-bp:ro
      - ./data/export-zup:/data/export-zup:ro
      - ./output:/data/output
      - ./config.yaml:/config/config.yaml
```

`config.yaml`:

```yaml
import_roots:
  - /data/export-bp
  - /data/export-zup
```

Для каждой конфигурации: **«+ Новая»** → указать **точное имя** из `Configuration.xml` → импорт из соответствующего пути.

## Конфигурация уже в базе, но нет выгрузки в слоте

Типичная ситуация после миграции или legacy-индексации: в таблице есть объекты (`in_database`), колонка «Выгрузка» — `нет в слоте`, кнопки **Обновить** / **Переиндексировать** неактивны.

1. **«+ Новая»**
2. В поле **Имя** введите имя **точно как в базе** (например `ЗарплатаИУправлениеПерсоналомКОРП`)
3. Импортируйте выгрузку (путь или ZIP)
4. **Проверить слот** → **Обновить** → **Переиндексировать**

## Этапы индексации и ожидаемое время

| Этап | Что происходит | Где смотреть прогресс |
|------|----------------|------------------------|
| 1. Объекты | Обход всех XML, сравнение hash, markdown | Лог задачи: `Обработано N/M объектов` |
| 2. Чанки | Разбиение markdown (без отдельных логов в UI) | — |
| 3. Эмбеддинги | Батчи в API провайдера | Лог: `Эмбеддинги: X/Y чанков (батч …)` |

Пока идёт **Переиндексация**, токены API тратятся только на этапе 3. Долгий **Обновить** на большой конфигурации (десятки тысяч объектов) — это этап 1, даже если все объекты «пропущены» по hash: XML всё равно читается для проверки.

После прерывания контейнера задача в памяти теряется, но данные на диске сохраняются. Обновите страницу UI (**Ctrl+F5**) — активная задача подхватится снова, если сервер ещё выполняет job.

## Слот: файлы vs привязка

| Режим | Содержимое `output/exports/{Имя}/` | Колонка «Выгрузка» |
|-------|--------------------------------------|--------------------|
| Привязка (по умолчанию) | Файл `.import_source` с абсолютным путём | `staging` |
| Копирование (`mirror: true`) | Полная копия XML | `в слоте` |
| ZIP-импорт | Полная копия из архива | `в слоте` |

Индексация всегда читает выгрузку через `slot_export_root()` — либо из слота, либо по пути из `.import_source`.

## HTTP API (per-configuration)

| Метод | Описание |
|-------|----------|
| `POST /configurations` | Создать запись / папку слота `{"name":"…"}` |
| `GET /configurations/{name}` | Карточка: `has_export`, `export_linked`, `embedding_status`, … |
| `POST /configurations/{name}/import` | Импорт ZIP (multipart) |
| `POST /configurations/{name}/import-path` | Импорт пути: `{"source":"…","mirror":false}` |
| `POST /configurations/{name}/detect` | Проверка слота и имени |
| `POST /configurations/{name}/index` | Индексация: `skip_embeddings`, `force`, `embeddings` |
| `POST /configurations/{name}/embed` | Только эмбеддинги: `force`, `embeddings` |
| `GET /configurations/jobs`, `GET /configurations/jobs/{id}` | Статус фоновых задач |
| `DELETE /configurations/{name}` | Удаление (`?async_job=true` для фона) |

Устаревшие: `POST /configurations/index`, `POST /upload`, `POST /reindex` — используйте API выше.

## CLI

```bash
# Метаданные из слота
conf-doc index --configuration Имя --skip-embeddings

# Эмбеддинги
conf-doc embed --configuration Имя

# Полный цикл
conf-doc index --configuration Имя

# Перенос legacy export_path в слоты
conf-doc configurations migrate-exports
```
